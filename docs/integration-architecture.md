# 集成架构草案

## 1. 文档目的

本文档用于定义 AI 虚拟团队工作台与外部协作资产系统、业务管理系统之间的集成边界、职责划分和落地原则。

本文档强调两点：

1. `Nextcloud` 和 `ERPNext` 是当前优先支持的首批参考实现。
2. 本系统不应与这两个产品形成不可替换的强耦合关系。

## 2. 总体定位

本系统不是要重做一个文件协作平台，也不是要重做一个完整 ERP，而是要成为一个以 AI 员工为核心的统一工作台。

系统职责分为三层：

1. 体验层
   本产品前端，负责团队、AI 员工、项目、坐台程序、客户会话和工单工作台体验。
2. AI 协作层
   本产品后端，负责身份、权限、上下文装配、Agent 编排、工具调用、任务路由和会话管理。
3. 外部能力层
   外部协作资产系统和业务管理系统提供文件、知识、客户、项目、工单等能力。

一句话定义：

本系统是“AI 虚拟团队工作台”，可对接协作资产系统和业务管理系统；`Nextcloud` 与 `ERPNext` 是首批优先适配对象。

## 3. 为什么不建议强耦合

如果系统与 `Nextcloud` 和 `ERPNext` 强耦合，会出现以下问题：

1. 产品价值容易被理解为这两个系统的配套层，而不是独立的 AI 工作台。
2. 用户一旦已有其它生态，迁移成本会明显提高。
3. 后续支持类似系统时，需要修改大量核心业务逻辑。
4. 商业化时会被底层产品选型强绑定。

因此更合理的方向是：

1. 产品层面优先支持 `Nextcloud` 和 `ERPNext`。
2. 架构层面与“能力类型”耦合，而不是与“具体品牌”耦合。

## 4. 外部能力分类

### 4.1 协作资产系统

这类系统主要承载协作资料和知识资产能力，包括：

1. 文件存储
2. 文档资料归档
3. 团队共享资料
4. 日历、联系人、任务等协作信息
5. AI 员工可检索的知识来源

这类系统更适合承担“资料层”和“知识层”角色。

首批优先适配对象：`Nextcloud`

### 4.2 业务管理系统

这类系统主要承载标准化业务对象和经营流程，包括：

1. 客户主数据
2. 线索和商机
3. 项目和任务
4. 工单或售后流程
5. 报价、订单、发票
6. 财务和履约相关记录

这类系统更适合承担“业务层”和“经营层”角色。

首批优先适配对象：`ERPNext`

## 5. 本系统职责

本系统应聚焦以下职责：

1. 用户登录与身份识别
2. 团队与成员管理
3. AI 员工创建与编排
4. 项目工作台体验
5. 坐台程序配置与对客入口
6. 客户会话管理
7. AI 驱动的工单生成与任务协作
8. 外部系统能力统一入口

换句话说，本系统负责“人和 AI 如何协作做事”，而不直接承担全部文件系统和 ERP 流程。

## 6. 领域边界定义

### 6.1 本系统主数据

建议本系统作为主数据源持有以下对象：

1. 用户 User
2. 团队 Team
3. 成员 Member
4. AI 员工配置 AIMemberProfile
5. 坐台程序 ConciergeApp
6. 会话 Conversation
7. Agent 执行日志
8. 工具调用日志
9. 集成连接配置

这些数据是本系统核心能力，不应依赖外部系统反向驱动。

### 6.2 本地视图加外部映射的数据

以下对象建议在本系统中保留轻量视图和映射关系，但不一定作为最终主数据源：

1. 项目 Project
2. 客户 Customer
3. 工单 Ticket
4. 任务 Task

推荐做法：

1. 本系统保留工作台展示和 AI 协作所需的轻量结构。
2. 本系统保留外部系统 ID 映射字段。
3. 根据集成深度决定最终以哪个下游系统为准。

### 6.3 优先以下游系统为准的数据

建议以下数据长期以业务管理系统为主：

1. 客户主档
2. 商机和销售流程
3. 报价、订单、发票
4. 正式项目管理对象
5. 财务相关记录

建议以下数据长期以协作资产系统为主：

1. 文件
2. 文档
3. 共享资料目录
4. 知识资产附件

## 7. 集成原则

### 7.1 单一权威来源原则

同一类核心能力尽量只有一个系统作为权威来源。

例如：

1. 文件正文和附件内容由协作资产系统持有。
2. 正式业务单据由业务管理系统持有。
3. AI 员工配置和会话上下文由本系统持有。

### 7.2 映射优先，不做全量复制

MVP 阶段不建议把外部系统对象全量复制到本系统数据库，而应优先采用“本地索引 + 外部 ID 映射”的方式。

### 7.3 工作台优先

本系统的价值不是代替下游系统，而是通过 AI 工作流和统一 UI 将多个系统能力整合成一个操作面。

### 7.4 Agent 安全边界

AI 员工访问外部系统时，必须经过权限控制、动作授权和操作日志记录，不能直接拥有无限写权限。

### 7.5 能力解耦原则

本系统应与“能力类型”耦合，而不是与“具体品牌”强耦合。

例如：

1. 本系统需要的是文件与知识能力，而不是只能接 `Nextcloud`。
2. 本系统需要的是客户、项目、工单能力，而不是只能接 `ERPNext`。

## 8. 推荐系统分工

### 8.1 本系统负责

1. 登录认证
2. 团队和成员
3. AI 员工
4. 坐台程序
5. 客户会话
6. Agent 编排
7. 工单生成逻辑
8. 统一工作台

### 8.2 协作资产系统负责

1. 文件库
2. 知识附件
3. 项目资料目录
4. 共享文档
5. 日历 / 联系人 / 任务等协作能力

首批优先适配对象：`Nextcloud`

### 8.3 业务管理系统负责

1. 客户主档
2. 销售线索
3. 商机与交易流程
4. 标准业务项目
5. 服务单和售后记录
6. 财务对象

首批优先适配对象：`ERPNext`

## 9. 适配器架构建议

### 9.1 总体建议

不建议在业务逻辑层直接耦合 `Nextcloud` 或 `ERPNext` 的 API，而应在本系统内部建立统一适配器层。

推荐结构：

1. 产品层和 Agent 层调用统一能力接口。
2. 适配器层负责将统一接口转为具体外部系统调用。
3. 每个外部系统通过独立 connector / adapter 接入。

### 9.2 推荐能力接口

建议优先抽象以下接口：

1. `FileKnowledgeProvider`
   提供文件、目录、知识检索能力。
2. `CustomerProvider`
   提供客户查询、客户详情能力。
3. `ProjectProvider`
   提供项目查询、项目映射能力。
4. `TicketProvider`
   提供工单创建、查询、状态同步能力。
5. `TaskProvider`
   提供任务创建、查询、状态更新能力。

### 9.3 首批适配器

第一阶段建议仅实现两个具体适配器：

1. `NextcloudAdapter`
   实现 `FileKnowledgeProvider`，可视情况扩展协作类能力。
2. `ERPNextAdapter`
   实现 `CustomerProvider`、`ProjectProvider`、`TicketProvider`、`TaskProvider`。

当前代码状态：

1. 已落地统一 provider 接口骨架
2. 已落地 `NextcloudAdapter` 和 `ErpNextAdapter`
3. 已注册到后端依赖注入
4. 已支持团队级连接配置、连接校验和预览接口
5. `NextcloudAdapter` 已尝试通过 `status.php` 与 WebDAV `PROPFIND` 进行真实连接校验和目录读取
6. `ErpNextAdapter` 已尝试通过 `frappe.auth.get_logged_user` 与 REST resource API 进行真实连接校验和对象预览
7. 当外部系统未配置、不可达或返回异常时，当前预览层会回退为占位结果，以保持工作台可演示

### 9.4 后续扩展方式

未来如果要支持类似生态，可以继续新增其它 adapter，而不需要改动上层 Agent 角色和业务流程定义。

因此推荐路径是：

1. 先把接口抽象做对。
2. 首批仅落地 `Nextcloud` 和 `ERPNext`。
3. 后续视客户需求扩展同类系统。

## 10. 建议集成方式

### 10.1 协作资产系统集成方式

建议优先集成以下能力：

1. 文件列表读取
2. 文件元数据读取
3. 文件内容索引
4. 指定目录作为项目知识库
5. 指定文件夹与项目绑定

MVP 目标：

让 AI 员工可以读取与项目相关的资料和知识资产。

首批优先落地对象：`Nextcloud`

### 10.2 业务管理系统集成方式

建议优先集成以下能力：

1. 客户查询
2. 项目查询
3. 工单创建
4. 任务创建
5. 客户与项目映射

MVP 目标：

让坐台程序和 AI 员工能够把客户会话沉淀成业务管理系统中的标准业务对象。

首批优先落地对象：`ERPNext`

## 11. 推荐数据映射方案

### 11.1 项目映射

本系统 `Project` 建议增加：

1. `source_type`
2. `source_id`
3. `external_ref`

示例：

1. `source_type = local`
2. `source_type = external_business_system`
3. `source_type = hybrid`

### 11.2 客户映射

本系统 `Customer` 建议保留：

1. 本地客户视图字段
2. `external_customer_id`
3. `external_system_type`

这样客户可以先由会话产生，再决定是否同步为正式客户。

### 11.3 工单映射

本系统 `Ticket` 建议保留：

1. 本地工单 ID
2. `external_ticket_id`
3. `external_system_type`
4. 来源会话 ID
5. 来源坐台程序 ID

这样本系统可以先承接 AI 工作流，再同步到业务管理系统。

### 11.4 知识资产映射

本系统建议不存储完整文件，只存：

1. `external_file_id`
2. `external_system_type`
3. 路径
4. 文件名
5. MIME 类型
6. 索引摘要
7. 向量索引引用

## 12. AI 员工如何调用外部系统

建议把外部系统能力封装成工具，而不是让 Agent 直接感知底层系统。

例如：

1. `search_project_files`
   从协作资产系统查询项目资料。
2. `get_customer_profile`
   从业务管理系统查询客户档案。
3. `create_service_ticket`
   在业务管理系统创建服务工单。
4. `create_project_task`
   在业务管理系统创建任务。

这样做有三个好处：

1. Agent 调用模型统一。
2. 权限控制集中。
3. 后续可以替换下游系统而不改 Agent 角色定义。

## 13. MVP 推荐集成范围

### 13.1 协作资产系统 MVP

1. 项目资料目录绑定
2. 文件读取
3. 文件内容检索
4. 提供给 AI 员工做知识上下文

首批优先落地对象：`Nextcloud`

### 13.2 业务管理系统 MVP

1. 客户查询
2. 项目查询
3. 工单创建
4. 工单状态同步

首批优先落地对象：`ERPNext`

MVP 不建议一开始就接入：

1. 财务模块
2. 采购模块
3. 库存模块
4. 全量 CRM 流程

## 14. 推荐落地顺序

### 阶段一：本系统基础能力

1. 用户与团队
2. AI 员工
3. 项目
4. 坐台程序
5. 会话
6. 本地工单

### 阶段二：协作资产系统集成

1. 项目资料目录映射
2. 文件读取与搜索
3. AI 知识检索接入

首批优先落地对象：`Nextcloud`

### 阶段三：业务管理系统集成

1. 客户映射
2. 项目映射
3. 工单同步

首批优先落地对象：`ERPNext`

### 阶段四：Agent 深度协作

1. AI 从协作资产系统取知识
2. AI 向业务管理系统写业务对象
3. AI 基于业务结果持续跟进

## 15. 关键风险

1. 如果本系统和业务管理系统都维护完整项目和客户数据，后续会出现双写冲突。
2. 如果 AI 直接写外部系统而没有审批边界，容易产生错误业务数据。
3. 如果协作资产系统文件权限没有正确映射到团队和项目，会造成知识越权。
4. 如果工单在本地和业务管理系统双向流转逻辑不清，会导致状态不一致。
5. 如果过早做成“万能连接器平台”，可能导致 MVP 落地速度明显下降。

## 16. 建议结论

最合理的落地方向不是把本系统做成另一个 `Nextcloud` 或 `ERPNext`，也不是只为这两个系统定制，而是：

1. 本系统负责“统一工作台 + AI 员工 + 客户入口 + 会话与协作编排”
2. 协作资产系统负责“协作资料和知识资产”
3. 业务管理系统负责“标准业务对象和经营流程”
4. `Nextcloud` 和 `ERPNext` 作为首批优先适配对象落地
5. 架构层通过 adapter / provider 接口保持可替换、可扩展

因此，本系统应该成为：

“连接 AI 虚拟团队、客户交互、知识资产和业务系统的统一操作层。”
